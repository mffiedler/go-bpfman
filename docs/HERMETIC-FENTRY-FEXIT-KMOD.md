# Hermetic fentry/fexit testing via a private kernel module

## Problem

The e2e suite has a class of fentry/fexit tests that attach BPF
programs to `do_unlinkat`, drive the kernel hook by issuing unlink
syscalls from a workload subprocess, then assert exact equality:

    counter == events * weight

Each test filters by TGID, so other processes' unlinks cannot
contaminate the count. The arithmetic is correct in isolation. The
suite still flakes with off-by-one shortfalls on the longest-lived
program in tests that share a hook with their siblings, e.g.
`mfx_a` reading `21*weight` when the test fired `22*weight`.

The miss is not in user space and not in the BPF program. It is
kernel-side event loss during a trampoline rebuild. fentry/fexit
attach goes through `bpf_trampoline_update`: a JITed image is
built containing every program currently attached to the target
function, then `text_poke_bp` swaps the function-entry patch.
Under suite-level contention -- where several tests in parallel
all attach to or detach from `do_unlinkat` -- there is a window
during which an in-flight call can pass through the function
without running every program in the rebuilt image. The exactly-
one shortfall is consistent with one such window per stressed
run.

The PID filter does not help: event loss happens before the BPF
program runs.

The current pragmatic fix on the table is a suite-level mutex
serialising every fentry/fexit test on `do_unlinkat`. It removes
one source of contention -- our own parallel tests -- but the
test remains non-hermetic against the rest of the host: any
system observability tool (Cilium, Pixie, Tracee, Falco), a
running `bpftrace` session, or a sysadmin probe that attaches to
`do_unlinkat` will reproduce the same flake.

## Goal

A test design where exact-equality fentry/fexit assertions hold
deterministically regardless of:

- how many of our own tests run in parallel,
- what other BPF tooling is loaded on the host,
- whether an interactive `bpftrace` session is running.

Plus, no per-test serialisation cost.

## Approach

Ship a small in-tree kernel module that exports a fixed pool of
private kernel functions reserved for the e2e suite. Each test
grabs an unused slot from a harness-side allocator at startup,
attaches its fentry/fexit programs to that slot's function, and
drives events by writing to a debugfs file backed by the same
slot. Slots are released on test end.

No production code path or third-party tool has any reason to
attach to a symbol named `bpfman_e2e_target_017`. Each slot has
its own trampoline, so concurrent attaches on different slots do
not contend. The workload subprocess and PID filter are no
longer needed for these tests: the trigger surface is private,
the test process is the only writer, and the slot's counter
program is the only reader.

## Module shape

Module name: `bpfman_e2e_targets`. License: GPL.

### Function pool

A fixed pool of `noinline notrace` functions, e.g.:

```c
noinline notrace long bpfman_e2e_target_000(unsigned long arg) { return arg; }
noinline notrace long bpfman_e2e_target_001(unsigned long arg) { return arg; }
...
noinline notrace long bpfman_e2e_target_031(unsigned long arg) { return arg; }
```

Generated by macro / token-pasting; one source file. Pool size
starts at 32; grows if a test ever needs more concurrent slots.
`noinline` so the compiler does not eliminate them. `notrace` so
they are not subject to ftrace dynamic patching independently of
the BPF trampoline. The function body is whatever the test wants
to observe; returning the input keeps the verifier happy and
gives the BPF program a return value to check.

### Trigger surface

Per-slot debugfs entries under `/sys/kernel/debug/bpfman_e2e/`:

    /sys/kernel/debug/bpfman_e2e/trigger_000
    /sys/kernel/debug/bpfman_e2e/trigger_001
    ...

A `write(2)` to `trigger_NNN` invokes `bpfman_e2e_target_NNN`
once per write call. Bytes written are parsed as the `arg`
value; an empty write is treated as `arg=0`. Mode `0600`,
owned by root, so unprivileged processes on a multi-tenant
host cannot fire our triggers.

### BTF requirement

`CONFIG_DEBUG_INFO_BTF_MODULES=y` must be set on the running
kernel for fentry/fexit to find the module's functions via BTF.
Standard on Fedora and recent Ubuntu. `RequireKmodTargets` test
gate refuses to run if absent.

### Module lifecycle

Loaded once per host (or per ephemeral CI VM) and left resident.
Idle cost: a few KB of kernel memory plus the symbol table
entries; no timers, no workqueues, no interrupts, no locks held.
TestMain in the e2e binary ensures the module is loaded
(`modprobe` if absent, accept if present); no rmmod on teardown.

A crashed test that left a `bpf_link` pinned to a slot's
function is harmless: the slot allocator simply will not hand
that index out until the pin is removed, and TestMain can
optionally sweep stale pins under our pin root before any test
runs.

## Slot allocation

Harness-side, in `e2e/helpers.go`:

- `acquireKmodSlot(t)` returns a `KmodSlot{Func, TriggerPath, Index}`
  and registers `t.Cleanup` to release.
- Free-list backed by a `chan int` of size `poolSize` populated at
  `TestMain` time.
- A test that needs more than one slot calls the helper N times.

If the pool is exhausted (more concurrent fentry/fexit tests than
slots), the helper blocks. We size the pool generously enough
that this never happens in practice.

## Test usage

A fentry/fexit test goes from:

```go
workload := startWorkload(t)
... attach to do_unlinkat ...
workload.Unlink(eventsPerWave)
```

To:

```go
slot := acquireKmodSlot(t)
... attach to slot.Func ...
slot.Fire(eventsPerWave)
```

`slot.Fire(n)` writes `n` newline-separated values to
`slot.TriggerPath`. The kmod runs `bpfman_e2e_target_NNN(arg)`
once per write. No subprocess, no NDJSON ack, no PID filter.

The BPF program no longer needs `expected_pid`; remove it from
the counter macro for the kmod-targeting variants. (The
`do_unlinkat`-targeting tests, if any remain, still need it.)

## Build plumbing

Add an `e2e/kmod/` directory:

    e2e/kmod/
      Makefile           # kbuild stub: obj-m += bpfman_e2e_targets.o
      bpfman_e2e_targets.c

Top-level Makefile gains:

    KMOD_DIR := e2e/kmod
    KMOD_KO  := $(KMOD_DIR)/bpfman_e2e_targets.ko

    $(KMOD_KO):
        $(MAKE) -C $(KMOD_DIR) KDIR=/lib/modules/$$(uname -r)/build

    kmod: $(KMOD_KO)

    test-e2e: $(BIN_DIR)/e2e.test $(KMOD_KO)
        sudo insmod $(KMOD_KO) || true   # idempotent
        ...

Cleaner: `test-e2e` recipe sources a small `hack/load-kmod.sh`
that does idempotent insmod with errcode interpretation
(EEXIST is fine; anything else fails the build).

## Deployment matrix

| Environment                          | Loadable? | Notes                                                              |
|--------------------------------------|-----------|--------------------------------------------------------------------|
| GH-hosted runner (`ubuntu-latest`)   | yes       | Ephemeral VM, root, headers via apt, module dies with VM.          |
| Self-hosted dedicated runner         | yes       | Persistent. Idempotent load is fine; no rmmod on teardown.         |
| Self-hosted shared runner            | risky     | Module visible to other tenants. Acceptable only with debugfs 0600.|
| Container-based (ARC, k8s)           | no        | Needs CAP\_SYS\_MODULE = privileged container; usually forbidden.    |
| Hardened / lockdown / secure boot    | no        | Unsigned modules rejected. Out of scope.                           |

The bpfman e2e suite already requires root, CAP\_BPF, CAP\_NET\_ADMIN,
BPF filesystem mounts, and netns manipulation. Adding "loads one
module from this repo" is an incremental capability ask, not a
category change, on environments where the suite can run at all.

## Where else this design helps

- **fentry / fexit**: primary customer.
- **LSM**: same trampoline mechanism. Future LSM tests can use
  the same slot scheme by adding `LSM_HOOK("file_open")`-style
  exports to the module if/when needed.
- **uprobe / uretprobe**: secondary. Uprobes key on
  `(inode, offset)`. The e2e binary's own symbols are shared
  across tests because every workload subprocess is the same
  inode. The kmod can host uprobe targets too via additional
  exported userspace-callable symbols, but the contention is
  lower-stakes (no trampoline rebuild) and the mutex / per-test
  uprobe stub may be sufficient. Defer until measured.

Hermetic by construction, no kmod required:

- TC / TCX / XDP (per-veth in private netns)
- kprobe / kretprobe / tracepoint / raw\_tracepoint (single
  `rcu_assign_pointer` publish, no rebuild window)
- cgroup-based and socket-based program types (per-cgroup,
  per-socket)

## Alternatives considered

1. **Suite-level mutex on `do_unlinkat`-touching tests.**
   Twenty-line change. Fixes the in-suite contention; does not
   fix the host-contention case. Cheap insurance, complementary
   to this design rather than competing with it.

2. **Per-test kernel function from a public-symbol pool.**
   Every test picks a different in-tree function (`do_renameat2`,
   `do_chmod`, etc.). Same hermetic property within the suite,
   but: (a) every BPF object and workload op needs a per-test
   variant; (b) the function pool is finite and shrinks with
   each new test; (c) still vulnerable to host-side BPF tooling
   that happens to attach to whichever public function we
   picked. Worse on every axis than the kmod for higher
   per-test code cost.

3. **Uprobe on a binary-private symbol.**
   Sidesteps trampolines entirely (different attach mechanism).
   But it does not exercise bpfman's fentry/fexit code path,
   which is the actual subject of these tests. Not a substitute.

## Non-goals

- Replacing the `do_unlinkat`-driven kprobe / kretprobe /
  tracepoint tests. Those are not affected by trampoline rebuild
  and do not need this machinery.
- Generating slots at runtime. Fixed pool, generated at compile
  time. Simpler module, simpler harness.
- Cross-architecture variation. Module is plain C; works on any
  arch the kernel supports BTF for. No assembly.

## Open questions

1. Pool size. Start at 32; revisit if any single test needs more
   than four concurrent slots.
2. Whether the trigger interface should be debugfs (simpler) or
   a `/dev/bpfman-e2e` chardev with an ioctl (slightly tighter
   permissions story, more code). Default to debugfs.
3. Whether the module should expose any symbols beyond the
   fentry/fexit targets -- e.g., a deliberately-noinline helper
   the targets call, for testing programs that need a non-leaf
   function. Out of scope for v1.
4. Whether to ship a signed variant for hardened environments.
   Out of scope; revisit if a customer asks.

## Rollout

1. Land the suite-level mutex first; the suite goes green
   immediately.
2. Land the kmod and harness changes behind a build tag /
   environment knob so the legacy `do_unlinkat` flow continues
   to work for environments where the kmod cannot load.
3. Migrate fentry/fexit tests to slot-based attach one by one;
   keep the legacy path until every test on it has been moved.
4. Remove the `do_unlinkat`-targeting fentry/fexit code, the
   workload subprocess for those tests, and the suite-level
   mutex.
