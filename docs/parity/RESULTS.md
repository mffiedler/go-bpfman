# Rust parity results

Generated by hand-driven proof runs on 2026-06-26. Every command was
run under `sudo`; Go used `--runtime-dir /run/bpfman-parity-go`, Rust
used its default `/run/bpfman`. See `ENVIRONMENT.md` for the full setup
and the interface recipe. Raw transcripts are under `outputs/`.

Binaries: Go `c392521a` (branch `doc-types`), Rust `bpfman 0.6.0`
(worktree `8e5a9d29`).

`DIVERGE` marks an observed behavioural difference on the same input,
not a verdict that Go is correct and Rust is wrong (or vice versa);
some divergences may be bugs on either side. See the calibration note
at the top of `ISSUES.md`.

"Lifecycle" means the full sequence was exercised on both
implementations and the semantic link/program state compared:

```
load -> get program -> attach -> get link -> link list -> program list
      -> detach -> unload -> get program (expect not-found)
```

`SEC("xdp/xdp_stats")`, which Rust rejects as an invalid XDP section

Go and Rust. The XDP baseline above uses `xdp_pass.bpf.o` (`SEC("xdp")`).

## Ordering chains

Contract: attaching to a dispatcher (XDP/TC) or a TCX hook orders the
chain by ascending priority, and each link's `position` is its rank.
For XDP/TC the Go `dispatcher get <type> <nsid> <ifindex>` command shows
the absolute chain order as ground truth (Rust has no dispatcher
command, so Rust is checked via per-link `position` only).

Attach order `50 10 40 20 30` -> expected positions `4 0 3 1 2`.

| Scenario | Go | Rust | Result | Transcripts |
| --- | --- | --- | --- | --- |
| XDP priority ordering (scrambled) | per-link `4 0 3 1 2`; `dispatcher get` order `pos0=10 .. pos4=50` | per-link `4 0 3 1 2` | PARITY (dispatcher-confirmed) | `ordering-xdp.*` |
| TC priority ordering (scrambled) | per-link `4 0 3 1 2`; `dispatcher get` order `pos0=10 .. pos4=50` | per-link `4 0 3 1 2` | PARITY (dispatcher-confirmed) | `ordering-tc.*` |
| TCX priority ordering (distinct progs) | per-link `4 0 3 1 2` | per-link `4 0 3 1 2` | PARITY (native mprog) | `ordering-tcx.*` |
| same program attached twice to a TCX hook | reject, exit 1 (`already attached ... detach it first`) | reject, exit 1 (kernel `bpf_mprog_attach failed ... File exists`) | both reject; error origin differs | `ordering-tcx.*` |
| equal-priority tie-break (3x prio 100) | deterministic chain `[#3,#1,#2]` (4/4 runs) | **non-deterministic** (`[#3,#2,#1]` and `[#3,#1,#2]` both seen over 8 runs) | **DIVERGE** (ISSUE-8) | `ordering-tiebreak.*` |
| detach from middle (attach 10,20,30; detach 20) | re-compacts `10:0 20:1 30:2` -> `10:0 30:1` | re-compacts `10:0 20:1 30:2` -> `10:0 30:1` | PARITY | `ordering-detach-middle.*` |
| slot reuse after detach (XDP) -- fill 10, detach prio 400, reattach prio 350 | members `10 -> 9 -> 10`; new link position `3` | links `10 -> 9 -> 10`; new link position `3` | PARITY | `slot-reuse-xdp.*` |
| slot reuse after detach (TC) | members `10 -> 9 -> 10`; new link position `3` | links `10 -> 9 -> 10`; new link position `3` | PARITY | `slot-reuse-tc.*` |
| slot exhaustion -- 11th attach to a full dispatcher (XDP/TC) | reject, exit 1 (`no free dispatcher slots (all 10 occupied)`) | reject, exit 1 (`No room to attach program. Please remove one and try again.`) | PARITY (both exit 1; wording differs) | `slot-exhaust-{xdp,tc}.*` |

Notes:
- XDP/TC allow the same program attached multiple times (the dispatcher
  chains links); TCX does not (native mprog rejects a duplicate prog on
  one hook), so the TCX chain uses distinct program loads.
- Both implementations place the most-recently-attached equal-priority
  link at position 0; they differ only in the relative order of the
  remaining equal-priority members (and Rust's is unstable).

The namespace setup recipe is recorded in `ENVIRONMENT.md`.

Note on the deleted-namespace case: after the namespace (and its
interface) is destroyed, both implementations still detach and unload
cleanly and leave no leftover bpfman state. Go's `link get` between the
deletion and the detach still reports `kernel_seen: true`; I could not
confirm whether that is accurate, because `bpftool link show` did not
enumerate that XDP-in-netns dispatcher link even before the deletion,
so the probe is inconclusive. It is *not* established that Go reports a
stale value -- in a controlled pin-removal test Go re-probes liveness
correctly (see `GO-AUDIT.md`). Recorded as unverified, not as a Go
defect.

## Representation differences (semantically equal, not failures)

- XDP/TC `proceed_on`: Go emits kernel action numbers (XDP `[2,31]` =
  pass+dispatcher_return; TC `[3,30]` = pipe+dispatcher_return); Rust
  emits names (`pass, dispatcher_return` / `pipe, dispatcher_return`).
- Kernel program-type label: for a loaded XDP program `bpftool prog
  show` reports the kernel type as `ext` (it is loaded as a freplace
  extension for the dispatcher), on both implementations. Go's
  `status.kernel.program_type` reports `extension`, matching the
  kernel; Rust prints `xdp` under a field labelled "Kernel Type", which
  does not match the kernel's `ext`. So this is not a wash: Go's
  kernel-type label is accurate and Rust's is not (assuming "Kernel
  Type" is meant to be the kernel's `BPF_PROG_TYPE`; if Rust intends the
  logical bpfman type, it is a labelling choice). See GO-AUDIT.md.
- tracepoint identity: Go splits `{group, name}`; Rust joins
  `group/name`.
- Output format: Go supports `-o json` on every verb; Rust `get`/`list`
  are text-only (ISSUE-5). All structured comparison above used Go JSON
  vs Rust text.
- Link IDs: Go assigns sequential IDs from a high base; Rust assigns
  large apparently-hashed IDs. Normalised away.
